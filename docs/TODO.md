- **Fix `just merge` on squash/rebase merges (issue #14).** `just merge` fast-forwards local `main` from the local branch SHA, which orphans `main` from `origin/main` whenever GitHub uses squash-merge or rebase-merge (different SHA, same diff). Next branch cut from local `main` then surfaces phantom merge conflicts on the new PR. Fix lives in `common.just`'s `merge:` recipe — detect drift via `git cherry origin/main <branch>` (patch-id comparison): if every branch commit's patch is already on `origin/main`, `reset --hard origin/main` and `branch -D`; if branch has true unmerged commits, refuse and tell user to rebase. Full reproduction, three-case decision table, and proposed recipe in the issue. Reproduced on real incident (`phileggel/VaultCompass` PR #11). High priority — silent corruption.

- **Pre-commit hook should skip gracefully on empty-stack repos (issue #15).** First-ever bootstrap commit (after `sync-config.sh` on a fresh `git init`) is rejected by the synced pre-commit hook because `check.py --fast` fails on absent `package.json` / `src-tauri/`. Worst possible UX moment — user's first impression nudges them toward `--no-verify`. Fix: extend `kit/scripts/check.py` to treat missing stack inputs as `SKIPPED` rather than `FAIL` (same spirit as existing skip rows for shellcheck/Prettier when not installed). Issue lists 4 alternative fixes; option 1 (graceful skip) generalises to partial-stack states during incremental scaffolding. Decide between option 1 (graceful skip) and option 2 (path-based skip for kit-only commits) before implementing. High priority — first-impression UX.

- **Evaluate kit behaviour on no-DB Tauri projects.** A new Tauri app may legitimately not need SQLite (or any database). The kit currently assumes SQLx is part of the stack in many places; identify the surface area before deciding on fixes. Three axes to audit:
  1. **Scripts** — no script failure observed in the field so far on a no-DB Tauri project (`kit/scripts/check.py:96` cleanly skips when `.sqlx/` is absent, which is likely why). Still worth a sweep across all `kit/scripts/*` and any DB-touching agent helper scripts to confirm they all skip-not-fail when `.sqlx/`, `migrations/`, or DB env vars are absent — and to catch any partial-stack edge case (e.g. `migrations/` exists but `.sqlx/` doesn't, or the reverse).
  2. **Agents/skills that give incorrect or misleading advice** — sweep `kit/agents/*.md` and `kit/skills/*/SKILL.md` for hard SQLx assumptions (e.g. backend reviewer expecting SQLx idioms, test-writer-backend defaulting to SQLx integration tests, contract skill assuming a DB boundary, dep-audit assuming sqlx in Cargo.toml). For each, decide: gate behind detection, or add an "if your project uses a DB" caveat.
  3. **Dead surface area** — what ships to a no-DB project that's pure noise? Likely candidates: `just prepare-sqlx` recipe in `kit/common.just`, SQLx-specific reviewer rules, SQLx-flavored test-writer templates. Decide per item: ship-and-skip-when-irrelevant, gate at sync time, or split into a separate opt-in module.

  Output of this evaluation should be a concrete fix list (each item one of: graceful-skip, doc-gate, sync-time-exclude, or accept-as-noise) — not the fixes themselves. Related to issue #15 (graceful skip on empty stacks) but distinct: #15 is about _empty_ stacks during bootstrap; this is about _deliberately partial_ stacks (Tauri-without-DB) that are a permanent state, not a transient one.
